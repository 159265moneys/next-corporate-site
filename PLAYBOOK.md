# 第1章: 思想 — 何を「良い」とするか

AIの出力品質は、モデルの能力より「発注者が良し悪しの基準を言語化できているか」で決まる。
以下がこのプロジェクトの美学基準。**AIへの指示に必ずこの言葉のまま含めること。**

1. **「無駄にかっこいいは、あればあるほど善」** — 動き・3D・空間移動は過剰なくらいが正解。
   AIは放っておくと「上品で控えめ」に寄る。この一文で振り切らせる。
2. **「骨格より肉付け」** — 構成が正しくても、1個のボックス・1本の線が素のままなら未完成。
   **「ただの線・基礎フォントだけの箇所は原則ゼロ」**を合言葉にする。
3. **「読めないものは存在しないのと同じ」** — 装飾のための薄グレー文字・細字は禁止。
   ゴースト数字のような飾りも「読めるグラフィック要素」に格上げする。
4. **「画像素材ゼロ」** — 図形と数式で描けるものは全部コードで描く。
   軽い・無限解像度・後から一行で修正可能。様式化(イラスト調・アイソメ調)を恐れない。
5. **「統一性のある進行」** — ページ全体を貫く1つの物語構造(このサイトでは
   「設計図が、稼働する。」)を最初に決め、全セクションをその進行に従属させる。

# 第2章: 技術スタック標準形

ビルドツール禁止。CDN+素のHTML/CSS/JSで構成する(AIが最も高品質に出力できる形式)。

```html
<script src="https://cdnjs.cloudflare.com/ajax/libs/three.js/r128/three.min.js"></script>
<script src="https://cdnjs.cloudflare.com/ajax/libs/gsap/3.12.5/gsap.min.js"></script>
<script src="https://cdnjs.cloudflare.com/ajax/libs/gsap/3.12.5/ScrollTrigger.min.js"></script>
<script src="https://cdn.jsdelivr.net/npm/lenis@1.1.13/dist/lenis.min.js"></script>
```

ファイル構成(役割分離を厳守させる):
- `index.html` — マークアップ+コピー+SVGアイコン(インライン)+OGP+SVGデータURI favicon
- `css/style.css` — デザインシステム全部
- `js/main.js` — オーケストレーション(Lenis/ローダー/カーソル/ScrollTrigger演出/エンジンIO管理)
- `js/scene.js` — 固定背景のWebGLシーン(window.NextSceneとして公開)
- `js/sections.js` — セクション専用エンジン群(radar/iso/terminal/citation/dashboard)。
  各エンジンは `{start, stop, resize, renderStatic}` のインターフェースで統一し、
  IntersectionObserverで非表示時に完全停止する

フォント: Google Fonts「Space Grotesk」(英数 500/600/700)+「Noto Sans JP」(500/600/700/900)。
**300などの細字ウェイトは読み込むこと自体を禁止。**

# 第3章: デザインシステム標準値(コピペ可)

```css
:root{
  --bg:     #FAFBFC;  /* ベース。ほぼ白 */
  --ink:    #1A2333;  /* 文字。青みの墨 */
  --line:   #C5D4E8;  /* 罫線・薄い構造線 */
  --panel:  #E8EEF6;  /* 面。アイスブルー */
  --accent: #2563EB;  /* 青。CTAと差し色のみ */
  --dark:   #0B0F1A;  /* 反転セクション用 */
}
```

**鉄の掟:**
- accent青の画面占有は**5%以下**。CTA・下線・小ラベル・点滅灯のみ。
  「青のサイト」は青を塗ることではなく、青を我慢することで成立する。
- font-weight体系: 本文500 / サブコピー600 / 英マイクロラベル700 / 見出し900。
- ゴースト数字(セクション番号の巨大アウトライン文字): stroke 2.6px以上+
  ink色の22%不透明度+薄グラデ塗り併用。ダークセクションでは白系34%。
- 仕切り線は単純な1px線を全廃。グラデフェード線か目盛りティック付き線。
- 全ボタン: グラデ塗り(accent→深青)+内側ハイライト+hoverで光沢スイープ+
  矢印マイクロモーション+focus-visibleリング。
- 全カード: 微グラデ背景+内外二重ボーダー+コーナー装飾+hoverリフト&微グロー+
  番号タグ(IND-01等)+英マイクロラベル。
- ダークセクションをページ中盤に1つだけ入れ、明→暗→明のリズムを作る。

ヒーロー(FV)は街頭広告基準:
- 見出し clamp(2.7rem, 8.6vw, 8.6rem)、weight 900、skewX(-8deg)、2段組+段オフセット
- 文字背後に白ラジアルグロー+斜めプレートを敷き、背景演出との干渉を物理的に断つ
- キーフレーズにSVGストロークアニメの下線スウォッシュ

# 第4章: モーション設計原則

## 4-1. 基盤
- Lenis慣性スクロール(duration 1.25, expo out)。イージングはexpo/power4系。
  bounce系・elastic系は安っぽくなるため禁止。
- カスタムカーソル: ドット+遅延リング(lerp 0.16)、CTAはマグネット吸着(変位×0.22)。
- ローダー: 方眼グリッド出現→ロゴをstroke-dashoffsetで線画アセンブル→%カウンター→開幕。

## 4-2. スクロール構造の鉄則(実地で確立したルール)
1. **縦スクロールと横移動演出は絶対に同時進行させない。**
   横に動くセクションは必ずピン留めし、縦入力を横進行に100%変換。
   横が完了してから縦を再開する。モバイルでも同じ(非ピンの縦積みフォールバックに
   逃げると「完了前に通過して見えない」事故になる)。
2. **ピン留めトリガーはページ順で先に生成する。**後続トリガーの位置計算に
   ピンの延長距離を織り込ませるため。生成順を乱すなら `ScrollTrigger.sort()`。
3. ピンはページ全長を伸ばす。1セクション+130%〜+260%を目安に、3〜4箇所まで。

## 4-3. 固定WebGL背景(空間移動の主軸)
- 全画面固定canvasにパーティクル約950粒+近傍ライン網。
- 疑似カールノイズ(sin/cos合成)で常時ドリフト。隊形遷移中は振幅3.4倍+呼吸。
- スクロール進行に9キーフレームのカメラワークをsmoothstep補間+lerp追従:
  ロール±5〜20°、オービット±11〜55°、ドリー±12、遷移点でFOVパンチ+10。
  マウスパララックスは1.5倍。「ぐわんぐわん動くが酔わない」の実測解がこの数値帯。
- クライマックス: パーティクルが意味のある形に収束する(このサイトでは
  「N」が左端15%へ→オフスクリーンcanvasで字形サンプリングした「ext.inc」の
  粒子62%が画面外からeaseOutBack(s=2.2)で飛来し「Next.inc」完成)。
  文字収束はオフスクリーンcanvasに文字を描いてピクセルサンプリングするのが定石。

## 4-4. セクション演出パターン集(全ブロック違うデザインにする時の引き出し)
| パターン | 中身 | 技術 |
|---|---|---|
| Blueprint | 設計図フローチャートが線画で組み上がり、完成後も光パルスが流れ続ける | SVG stroke-dashoffset+CSSアニメ、横ピン |
| Conveyor | 書類カードがベルト流動、ゲート通過で「AUTO✓」スタンプ | CSS 3D transform、ピン+スクラブ |
| Radar | ダーク反転。掃引レーダー+候補ドットのピング波紋+タイマー | Canvas 2D |
| Terminal | 自動タイピング→フォーム自動入力→送信カウンター | DOM+setInterval |
| Isometric | Three.js実3Dジオラマ(詳細は4-5) | Three.js第2レンダラー |
| Citation | AI回答UIモックが回答をタイプし引用元が点灯、キーワードノード軌道回転 | DOM+Canvas |
| Dashboard | 折れ線/棒グラフの自己描画+KPIカウントアップ+A/Bカード定置クロスフェード | Canvas 2D |

## 4-5. 3Dジオラマの品質基準(「ちゃっちい」と言われないために)
- Canvas 2Dの箱組みはNG。Three.js+OrthographicCameraの正投影アイソメ。
- HemisphereLight+DirectionalLight+PCFSoftShadowMap(2048)の柔影。地面は方眼テクスチャ。
- **各ジオラマに常時マイクロアニメーションを最低2つ**(クレーン旋回・トラック周回走行・
  サーチライト光錐の掃引・LED明滅など)。「生きている街」にする。
- スクロールで主役化ツアー: ピンしてカメラが各ジオラマへパン+ズーム(1.8〜2.1x)、
  非主役はDIM色へlerp。関連カードと `.is-live` ハイライト同期。
- **ラベルは3D空間や図形の上に描かない。**下部の固定グリッドにHTMLチップを置き、
  world→screen射影のSVGコネクタ線で結ぶ(getBoundingClientRect基準で毎フレーム追従)。
- DPR上限2(モバイル1.5)、モバイルは影/AA/エッジオフ+ジオメトリ簡略化。

# 第5章: 地雷リスト(実際に踏んだバグと恒久対策)

**この章をAIに渡さないと、同じ穴に必ず落ちる。**

1. **バックグラウンドタブで開くとrAFが凍結し、GSAP駆動のローダーが永久に固まる**
   (overflow:hidden+Lenis停止が残りスクロール不能)。
   → ローダーに`setTimeout(7000)`のウォッチドッグを入れ、`tl.progress(1)`で強制完了。
   visibilitychangeでも進行0.05未満なら強制完了。**rAF依存の初期化には必ず
   setTimeout系のフェイルセーフを併設する**のが普遍原則。
2. **スクランブルテキストが乱れた文字のまま間欠的に固まる。**
   → rAFループ完走に依存せず、`setTimeout(dur+450ms)`で必ず最終文字列を書き込む
   finish()を併設+要素ごとのトークンで多重起動を無効化。
3. **innerHTML書き換えでアンカーのclickリスナーが消滅**(装飾のためのラップ処理が原因)。
   → クリック処理は個別バインドせずdocumentへのイベント委譲
   (`e.target.closest('[data-scrollto]')`)で書く。
4. **lenis.scrollToはisStopped中は黙って無視する。**
   → 実行前に`if (lenis.isStopped) lenis.start()`+300ms後にscrollYが動いていなければ
   `scrollIntoView`にフォールバック(非表示タブではbehavior:'auto')。
5. **ScrollTrigger.normalizeScroll(true)はプログラムスクロール(アンカー遷移)と干渉して
   更新停止することがある。**→ 不使用。タッチはネイティブスクロール+pinType:'fixed'既定+
   `ScrollTrigger.config({ignoreMobileResize:true})`で安定させる。
6. **iOS Safariのアドレスバー伸縮がresizeを連発しピンが暴れる。**
   → タッチ端末では幅が変わらない限りresize処理を無視する。
7. **ピンより先に後続トリガーを作ると位置計算がピン距離を含まない。**→ 4-2の生成順+sort。
8. **ヘッドレス/プレビュー環境はwindow.innerWidthが0を返すことがある**(実Chromeでは正常)。
   検証は必ず実ブラウザで。自動化タブはdocument.hidden=trueでrAF凍結するため、
   スクリーンショットが真っ白でも実環境のバグとは限らない。
9. CDN失敗時フォールバック: gsap/three未定義でも全コンテンツが静的表示される分岐を必ず用意。
   reduced-motionでは静止1フレーム描画(renderStatic)。

# 第6章: AIの操縦法 — 指示の書き方

## 6-1. 発注テンプレート(穴埋めして使う)

```
あなたは超一流のクリエイティブデベロッパーです。〈場所〉に〈サイト名〉を
フルスクラッチで構築してください。妥協なしの最高傑作を作ること。

# コンセプト『〈進行の物語を一言で〉』
# デザインシステム: (第3章の:rootと鉄の掟をそのまま貼る)
# 技術構成: (第2章をそのまま貼る)
# セクション構成とコピー(コピーは一字一句このまま使用):
  (見出し・サブコピーを全部確定させてから渡す。AIに書かせる場合は
   先にコピーだけ出させて人間が確定→実装は別発注)
# モーション品質基準: (第4章から該当部を貼る)
# 地雷対策: (第5章をそのまま貼る)
# 完了要件: node --checkで構文検証 / ローカル配信確認 / git commit+push /
  実装内容・行数・懸念点を報告
```

**コツ: 「何を作るか」だけでなく「何をしたら失格か」を書く。**
(bounce禁止・細字禁止・青5%超え禁止・ラベル被り禁止・素の1px線禁止)

## 6-2. フィードバックの語彙(短い言葉ほど効く)

| 言いたいこと | そのまま使うフレーズ |
|---|---|
| 動きが足りない | 「無駄なレベルで動きモリモリ」「ぐわんぐわん動かして」 |
| 密度が足りない | 「骨格は上々、肉付けが足りない」「1個のボックスさえも丁寧にこだわって」 |
| 安っぽい | 「ちゃっちい。10倍クオリティ上げて。あらゆる手段を使っていい」 |
| 読みにくい | 「視認性上げまくって。街頭広告くらいド派手に」「全部の文字を倍以上太く」 |
| UX事故 | 現象を具体的に(「スマホで横スライドが完了する前に下に行っちゃう」)→
  AIに「ルール化して全ブロック適用」まで求める |

**重要: 褒めるところは褒める(「最高!!」「ここは美しい」)。**残す部分を明示しないと
AIは全部作り直して良い部分まで壊す。

## 6-3. 反復ループの回し方
1. 一発目は仕様全部盛りで発注(部分発注より全体発注のほうが統一感が出る)
2. 出てきたものを**実ブラウザで見る**(スクショをAIに貼り戻すと精度が跳ね上がる)
3. 不満は1メッセージに複数まとめてよいが、各項目を具体的に
4. 2〜3往復で完成に収束させる。4往復超えたら仕様の言語化不足を疑い、
   このプレイブックの該当章を貼り直す
5. 修正のたびに `git commit+push` させる(戻れる状態を維持)

## 6-4. モデル非依存にするための注意
- このプレイブックの**数値をそのまま渡す**(「いい感じに」はモデル差が出る。
  「stroke 2.6px+ink22%」はモデル差が出ない)
- 能力が低めのモデルには: ファイルを1つずつ発注(まずcss→次にjs)、
  1セクションずつ発注、動くコードを渡して「この様式で次を書け」と模倣させる
- 検証コマンド(`node --check js/*.js`、ローカルサーバ起動確認)を必ず完了条件に含める
- 巨大出力が途切れるモデルには「続きから」と言わず、ファイル単位で再発注する
